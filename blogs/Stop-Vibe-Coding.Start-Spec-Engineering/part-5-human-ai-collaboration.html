Human + AI — The New Collaboration Model

Part 5 of 6

The AI writes the code. The human writes the architecture. Neither does it alone.

Every conversation about AI-assisted development arrives at the same question: what happens to software engineers?

Wrong question. The right question: in a world where AI can generate code from a design, what is the most valuable thing a software engineer can do?

The answer is exactly what engineers were always supposed to be doing: making good architectural decisions.

The Role Shift

If a well-specified design generates production-quality implementation code, then the scarce resource is no longer code — it's design. That's not deskilling engineering. It's a promotion of the most valuable engineering work.

The Collaboration Lifecycle

Product Team          Architect (Human)          AI Agent
─────────────         ──────────────────          ──────────────
Write User Stories →  Author/Review GASD     →  Generate Code
                      Make design decisions       Generate Tests
                      Resolve QUESTION blocks     Generate Docs
                      APPROVE the spec            Attach EVIDENCE
                                                  Open PR
                      Review Evidence         ←  Submit for merge
                      Merge or request changes

Stage 2 — Architect Defines How: The architect authors the GASD spec. They are **the authority on every design decision**: authentication strategy, error handling, type contracts, and global invariants.

Stage 4 — Blocking Ambiguity Surfaces: When an agent hits a gap, it raises a QUESTION block instead of guessing. Human decides. Spec updated. Decision permanently recorded.

Stage 6 — Evidence Embedded: When the agent finishes, it attaches an EVIDENCE block with test coverage, passthrough rates, and validation dates directly into the spec.

GASD as Institutional Memory

One of the most corrosive problems: architectural decisions made by engineers who've left become invisible. GASD makes institutional memory explicit. Every decision is recorded, dated, attributed, and carries its RATIONALE in version control.

The Two Fears — Answered

"Will GASD replace developers?" No. It replaces routine implementation work. It raises the value of trade-off analysis and architectural constraint-setting.

"Will juniors still learn?" Yes — arguably more. When decisions are explicit and reasoned in the spec, junior developers can read them, challenge them, and propose alternatives. That's how junior engineers become architects.

The New Measure of Senior Engineering

In a GASD-driven team, output is measured in quality of design decisions: how well did the spec eliminate ambiguity? How thoroughly do DECISION blocks capture trade-offs? How precisely do TYPE contracts prevent misuse?

GASD turns one-time AI conversations into permanent architectural contracts. It makes senior engineers the design authorities they always should have been.
